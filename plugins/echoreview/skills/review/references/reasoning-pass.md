# Reasoning pass

Phase 3 of `echo-review`. An open-ended pass over the diff to catch issues the pattern review (Phase 2) couldn't structurally express — contradictions between this diff and existing code, unjustified additions, missing-error-handling gaps. The pass is calibrated to surface fewer, higher-confidence findings.

See [`reasoning-examples.md`](./reasoning-examples.md) for the calibrated worked examples.

---

## Hard rules

1. **Confidence threshold: ≥80 to surface a finding.** Below 80, drop silently. There is no "low-confidence findings" output bucket.
2. **Severity restricted to `blocker` or `warning` only.** `suggestion` and `note` are reserved for Phase 2 (pattern review). If a reasoning-pass finding feels like a suggestion, that's a signal it doesn't belong in this pass.
3. **No hedging language.** From the prompt: *"If you would have to hedge with 'possibly', 'could be', or 'might', drop the finding. False positives erode trust faster than missed bugs."*
4. **Output every finding with an explicit numeric confidence value** in your internal notes (the value doesn't appear in the posted comment — see [`comment-template.md`](./comment-template.md)).

---

## What this pass is for

Three shapes of finding work well here:

- **Contradictions** — the diff conflicts with code or invariants elsewhere in the file or the PR. *"Function A assumes the cache is warm; this PR introduces a path that calls A before the cache populates."*
- **Unjustified code** — a new code path that, given the rest of the diff, has no obvious caller, no test, and no documented reason to exist. *"This new helper isn't called anywhere in the diff; either it's premature, or a call site got dropped."*
- **Missing error handling** — an operation that can fail (network, file, parse, external service) added without any failure path, in a context where the surrounding code does handle failure. *"Every other call to the gateway client in this file catches `GatewayTimeout`; this new one doesn't."*

If a finding doesn't fit one of these shapes, it almost certainly belongs in Phase 2 (universal floor or `.echoreview/patterns.md`), not here.

---

## Skip conditions

Phase 3 is a no-op if **any** of the following are true:

- **Lockfile-only PR**: every changed file matches one of
  `**/package-lock.json`, `**/yarn.lock`, `**/pnpm-lock.yaml`, `**/Cargo.lock`,
  `**/poetry.lock`, `**/Pipfile.lock`, `**/Gemfile.lock`, `**/composer.lock`,
  `**/go.sum`.
- **Docs-only PR**: every changed file matches one of
  `**/*.md`, `**/*.mdx`, `docs/**`, `README*`, `CHANGELOG*`, `LICENSE*`.
- **Generated-only PR**: every changed file's first 50 lines contain one of
  `@generated`, `DO NOT EDIT`, `autogenerated`, or `Code generated by`
  (case-insensitive).
- **Pure dependency version bump**: only manifest files changed
  (`package.json`, `Cargo.toml`, `pyproject.toml`, `requirements.txt`,
  `go.mod`, `Gemfile`, `composer.json`), and the diff modifies only version
  strings (no new deps, no removed deps, no script changes).

These are the only allowed skip conditions for v0.1. Team-configurable
skip patterns (`.echoreview/config.yml`) ship in v0.2+.

## Calibration

Anchor your confidence values against the four worked examples in
[`reasoning-examples.md`](./reasoning-examples.md). The mapping is:

- **95** — the contradiction is structural and the evidence is in the diff itself. No interpretation needed.
- **85** — the gap is concrete (missing handler, dropped caller) and you can point to the surrounding code that makes it a gap. Some interpretation, but not a stretch.
- **Below 80** — drop.

If your gut says "this is probably wrong but I'm not certain" — that's not 80. Drop it.

---

## Output shape (internal — for the skill to consume before formatting)

Each surviving finding:

```json
{
  "file": "src/foo.ts",
  "line": 42,
  "severity": "warning",
  "shape": "contradiction" | "unjustified-code" | "missing-error-handling",
  "confidence": 88,
  "rationale": "One short paragraph explaining the contradiction or gap. Used for your own bookkeeping; does NOT appear in the posted comment.",
  "comment": "The 1–2 sentence comment that will be posted, per comment-template.md."
}
```

The skill validates each finding against the diff position map and posts comments that survive. Confidence and rationale are not exposed in the GitHub UI.
